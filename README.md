# Snek — Runtime (open source)

> **This repo contains only the runtime component of Snek.** The Snek obfuscator and service are at [snekbite.com](https://snekbite.com). This code is published so you can audit exactly what executes on your machine inside a Snek-protected binary.

---

The open-source C++ runtime that is compiled into every [Snek](https://snekbite.com)-protected Python binary. The obfuscator that *produces* those binaries is closed-source; the runtime that *executes* them is not.

## What this is

When Snek protects a Python script, it produces a native binary containing:

1. Your Python bytecode, encrypted per-function with ChaCha20-Poly1305
2. A per-binary key (generated and embedded by the obfuscator)
3. **This runtime** — compiled and statically linked in

At execution, the binary initializes CPython, installs a frame evaluation hook (via [PEP 523](https://peps.python.org/pep-0523/)), and runs your code. The hook intercepts each function call, decrypts that function's bytecode on demand, and passes it to CPython's normal evaluator. Bytecode is never written to disk in decrypted form.

## Files

| File | Description |
|------|-------------|
| `runtime.cpp` | Frame evaluation hook — intercepts CPython function calls and triggers decryption |
| `runtime.hpp` | Public API: `install_frame_hook()`, `register_encrypted_blob()` |
| `decryptor.cpp` | ChaCha20-Poly1305 decryption via libsodium, bytecode unmarshaling via `PyMarshal_ReadObjectFromString` |
| `decryptor.hpp` | `decrypt_code_object()` signature |

The encryption key and the encrypted bytecode blobs are generated by the closed-source obfuscator and embedded in the binary at build time. This repo contains only the code that uses them at runtime.

## How the frame eval hook works

CPython (since 3.6, formalized in PEP 523) exposes a per-interpreter hook that replaces the default frame evaluator:

```c
_PyInterpreterState_SetEvalFrameFunc(interp, protected_eval_frame);
```

Every time CPython would normally call `_PyEval_EvalFrameDefault` to execute a function, it calls `protected_eval_frame` instead. The hook:

1. Reads the `PyCodeObject*` from the frame
2. Looks it up in a thread-safe cache of encrypted blobs
3. If found and not yet decrypted: calls `decrypt_code_object()`, stores the result
4. Swaps the frame's code object pointer to the decrypted `PyCodeObject*`
5. Resets the instruction pointer (`prev_instr`) to the start of the new code object
6. Delegates to `_PyEval_EvalFrameDefault` as normal

The decrypted code object lives in memory for the lifetime of the process. It is never serialized or written to disk.

### Python version differences

The internal layout of `_PyInterpreterFrame` changed significantly in Python 3.11. The runtime handles both:

- **Python 3.11+**: uses `_PyInterpreterFrame` with `f_code` at offset `0x20` and `prev_instr` at `0x28`; resets `prev_instr` when swapping code objects (required for the adaptive interpreter)
- **Python 3.10 and earlier**: uses `PyFrameObject` directly; `f_code` is a named struct member

## How the decryptor works

Each encrypted blob has the layout:

```
[ nonce: 12 bytes ][ tag: 16 bytes ][ ciphertext: N bytes ]
```

Decryption uses libsodium's `crypto_aead_chacha20poly1305_decrypt_detached`. On success, the plaintext is a valid marshaled `PyCodeObject`, deserialized with `PyMarshal_ReadObjectFromString`.

Symbol resolution for both `PyMarshal_ReadObjectFromString` and the frame eval API uses `dlsym(RTLD_DEFAULT, ...)` on Linux and `GetProcAddress(GetModuleHandleA(NULL), ...)` on Windows — no hardcoded library paths.

## Anti-debug checks

The binary runs two lightweight checks before initializing Python:

- **Linux**: reads `/proc/self/status` and checks `TracerPid` is `0`
- **Windows**: calls `IsDebuggerPresent()`
- Both: checks the `_` environment variable for `gdb`, `lldb`, `strace`, `ltrace`, `valgrind`

If a debugger is detected the binary exits silently before any decryption occurs.

## Dependencies

- **libsodium** ≥ 1.0.18 — ChaCha20-Poly1305 AEAD
- **CPython** 3.12 or 3.14 development headers (`python3.12-dev` / `python3.14-dev`)
- C++17 compiler (GCC or MinGW-w64 for Windows cross-compilation)

- The DRM / license activation system used by Pro binaries
- The key generation and embedding mechanism
